## 1. HIGH-LEVEL SYSTEM OVERVIEW
---
The "AIOps Server Log Anomaly Detector" is an automated, Machine Learning-powered tool designed for System Administrators and DevOps teams. Its purpose is to ingest raw, unstructured server logs, automatically parse them into readable data, and identify hidden issues (anomalies) like rare system crashes or severe performance bottlenecks. 

It eliminates the need for humans to manually read through thousands of log lines by automatically surfacing the critical issues and providing a "Runbook" of steps to fix them.

---
## 2. THE LOGICAL FLOW OF THE APPLICATION
---
When a user uses the application, the data flows through four main stages:

1. **Ingestion:** The user uploads a raw `.txt`, `.log`, or `.csv` file via the Streamlit web interface.
2. **Parsing (Structuring the Unstructured):** The raw text is fed into a parser that uses "Regular Expressions" (Regex) to extract meaningful fields out of the text string—specifically: the Timestamp, the Message, the HTTP Status Code, and the Latency (response time).
3. **Detection (The AI Brain):** The structured data is passed to the Detection engine. The engine applies both Unsupervised Machine Learning (to find weird, rare text messages) and Statistical Math (to find unusually slow response times).
4. **Visualization & Remediation:** The UI displays a dashboard with the total processed logs vs. flagged anomalies. It presents a clean table of only the critical errors and reads the reasons to generate an automated "DevOps Troubleshooting Runbook" telling the user exactly what to do next.

## 3. EXPLANATION OF EVERY FILE & ITS LOGIC
---

### A. `app.py` (The User Interface & Orchestrator)
- **Role:** This is the main script that runs the Streamlit web server. It connects all the pieces of the puzzle together.
- **Logic:** 
  - It creates the webpage styling (CSS), title, and file upload widget.
  - When a file is uploaded, it hands the file to `LogParser` (from `parser.py`).
  - If parsing is successful, it hands the data to `AnomalyDetector` (from `detector.py`).
  - Finally, it calculates metrics (Total Logs vs. Anomalies) and displays the isolated anomaly data on the screen alongside the runbook steps fetched from `runbook.py`.

### B. `src/parser.py` (The Data Extraction Engine)
- **Role:** Turns messy, raw text logs into a neat, organized table (Pandas DataFrame).
- **Logic:** 
  - It uses `re` (Python's Regular Expression library) to define specific patterns. 
  - It tries to match every single line of the uploaded text file against 3 predefined patterns:
    1. A standard Nginx/Apache web server log format.
    2. A typical application log format (e.g., Date Time [INFO] message).
    3. A generic fallback for other formats.
  - When it finds a match, it extracts the variables (status code, latency, message) and stores them cleanly. If it can't match a line, it just saves the whole line as a generic message so no data is lost.

### C. `src/detector.py` (The Machine Learning & Math Engine)
- **Role:** This is the core intelligence of the app. It flags which logs are normal and which are anomalies.
- **Logic:** It uses three distinct strategies to find errors:
  1. **Unsupervised ML (Text Clustering):** 
     - It uses `TfidfVectorizer` to convert text messages into a mathematical matrix of numbers based on word frequency.
     - It uses `KMeans Clustering` (set to 2 clusters) to group the logs. Because anomalies (crashes, fatal errors) are rare compared to normal traffic (successful logins), the algorithm will naturally group the thousands of normal logs into one large cluster, and the few rare errors into a tiny cluster. The code automatically flags the smaller cluster as the "Text Anomaly".
  2. **Statistical Analysis (The 3-Sigma Rule):** 
     - It looks at the "Latency" (speed) column. It calculates the average (mean) speed and the Standard Deviation (how much variance there is). 
     - It uses the "3-Sigma Rule" from statistics: Any log that takes longer than `Mean + (3 * Standard Deviation)` is mathematically considered an extreme outlier and is flagged as a "High Latency" anomaly.
  3. **Hardcoded Rules:** 
     - Any HTTP status code 500 or above is instantly flagged as a "Server Error".

### D. `src/runbook.py` (The Automated DevOps Assistant)
- **Role:** Provides actionable advice based on what the detector found.
- **Logic:** It contains a simple function `get_runbook_actions()`. It looks at the specific `anomaly_reason` generated by the detector (e.g., Did it fail because of text patterns? Latency? Status code?). Based on keywords in the reason, it returns a list of human-readable steps (like "Check database connection pool" or "Check disk space") to display on the dashboard.

## 5. TIPS FOR YOUR PRESENTATION Q&A
---
- **If they ask:** "Why did you use K-Means Clustering?"
  **Your Answer:** "I used K-Means because log anomaly detection is typically an Unsupervised Learning problem. In the real world, we rarely have perfectly labeled data showing what is an error and what isn't. K-Means allows the system to autonomously group the data based on text similarity, and since anomalies are inherently rare, the smaller cluster mathematically represents the outliers."

- **If they ask:** "What is TF-IDF?"
  **Your Answer:** "It stands for Term Frequency-Inverse Document Frequency. It's a natural language processing technique that converts log text into numbers. It gives high weight to unique, rare words (like 'StackOverflow') and low weight to common words (like 'INFO'), which perfectly prepares the data for the clustering algorithm."

- **If they ask:** "How does the latency detection work?"
  **Your Answer:** "It relies on the Empirical Rule of statistics (the 68-95-99.7 rule). By calculating the standard deviation of all response times, the system dynamically sets a threshold at 3 standard deviations above the mean. Anything past this threshold represents the top 0.3% of slowest requests, dynamically adapting to the server's baseline performance without needing hardcoded limits."
